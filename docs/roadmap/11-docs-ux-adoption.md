# Module 11 — Documentation, UX, and Adoption

> Part of the time series econometrics library roadmap. Master plan: [ROADMAP.md](../../ROADMAP.md).

**This module owns everything between the numerics and the user: the competitive positioning that tells us where to aim, the unified estimator grammar and results-object design, the documentation architecture (Diataxis quadrants plus a fifth pillar, the executable replication gallery that doubles as the validation suite), model cards, the "which model when" decision guide, migration guides, error messages that teach, plotting conventions, bundled canonical datasets, reproducibility stamping, benchmark-driven and correctness-driven marketing, naming, governance, and the publication strategy. The thesis is that in econometrics software, correctness that users can verify and documentation that makes decisions — not raw speed — is what converts the skeptical senior economist, and that these properties are cheap to mandate early and impossible to retrofit.**

## Purpose and scope

This module covers the non-numerical surface of the library. It has three layers. First, the **competitive audit**: a feature-by-feature map of every incumbent the library will be judged against — statsmodels and arch in Python, the Hyndman and Pfaff ecosystems in R, BEAR and the VAR Toolbox in Matlab, Stata's time series suite, and the commercial GUI packages — with explicit "learn from this, note where it falls short, and validate against this" verdicts per package. The audit is retained in full because it is the evidence base for where the library is most differentiated: modern macro econometrics (LP, modern SVAR identification, hierarchical BVAR, nowcasting) has no maintained Python home, and no library in any language tells users which model to use when.

Second, the **user experience contract**: one estimator grammar (immutable Spec → `fit()`/`sample()` → rich immutable Results) locked by RFC before v0.1 and enforced by contract tests; sensible loud defaults with citations; publication-ready output (LaTeX tables compiled in CI, IRF panels, fan charts); a typed warning/exception taxonomy whose messages teach; and dataframe-native interoperability (pandas and polars in, caller's type out). Third, the **adoption machinery**: Diataxis-structured docs executed in CI, model cards for every estimator, the replication gallery, migration guides from statsmodels/R/Stata, bundled canonical datasets so every example starts from one line of code, the public benchmark and validation-badge programs, JOSS/JSS papers, and multi-maintainer governance.

Its users are everyone: the applied macroeconomist deciding whether to trust the library, the grad student learning from the decision guide, the central-bank desk migrating from BEAR, the contributor reading the grammar RFC, and — increasingly — the AI coding assistant reading `llms.txt`. Relative to other modules, Module 11 defines the contracts (grammar, results objects, model-card schema, defaults registry, warning taxonomy, plotting style) that every estimator-owning module implements, and consumes their numerics to build the gallery, benchmarks, and parity matrix.

## Where existing tools fall short

- **statsmodels.tsa** is maintenance-constrained and partially stale: the VAR/SVAR module is frozen circa 2015 (AB-model only, no sign/zero/narrative/proxy identification, asymptotic or simple-MC IRF bands only), Kalman-based models are too slow for serious bootstrap or Monte Carlo work (a 10k-replication SARIMAX bootstrap takes hours), and it has no LP, no BVAR, no MIDAS, no multivariate GARCH, and multi-year PR queues.
- **arch** is the Python quality bar (validation against published numbers, robust SEs) but deliberately univariate — no DCC/BEKK/GO-GARCH is maintained anywhere in Python — and its bootstrap module goes undiscovered.
- **The Nixtla stack** proved benchmark-driven adoption works but is point-forecast only: no standard errors, no tests, no identification, recipe-only docs with zero theory, and its benchmark comparisons drew accusations of unequal convergence settings.
- **pmdarima** is wildly downloaded but effectively unmaintained and never matched R `forecast::auto.arima` — a ready-made migration-guide target.
- **darts, sktime, aeon** have the strongest existing thinking on unified grammars and backtesting ergonomics, but are econometrically shallow (no inference at all), and sktime's API churn plus the 2023 aeon governance fork are cautionary tales.
- **The R ecosystem is fragmented and fragile**: 15+ packages (vars, urca, svars, BVAR, bsvars, lpirfs, midasr, rugarch/rmgarch, tsDyn, strucchange) with incompatible APIs and results objects, mostly single-maintainer, several with CRAN-archival scares. No unified grammar exists anywhere in econometrics, in any language.
- **Macro identification lives in Matlab zips**, not packages: the VAR Toolbox, BEAR (Matlab-licensed, Excel-driven config), and Kilian-Lütkepohl textbook code — exactly the audience a `pip install` with the same output set wins.
- **No library documents which model to use when**: all incumbents ship reference docs; decision guidance lives in textbooks disconnected from code (fpp3/fable is the lone partial exception, and only for forecasting).
- **Cross-package numerical parity is undocumented everywhere**: users discover that Stata, R, and Python give different Johansen or Newey-West numbers with no explanation; nobody publishes defaults-difference tables.
- **Reproducibility tooling is absent**: no incumbent stamps seeds/versions/data hashes into results or exports journal-grade replication bundles, despite AEA Data Editor-era requirements.
- **Publication output is an afterthought**: no Python econometrics library produces journal-quality multi-model LaTeX tables (R's modelsummary/stargazer have no true Python peer) or standard IRF-panel and fan-chart figures out of the box.
- **Error messages across all incumbents are diagnostic dead-ends** — convergence failures and singular matrices with no guidance — a systematic teaching-UX gap.
- **Seasonal adjustment in Python has needless friction**: statsmodels requires a manual X-13 binary install; R's x13binary proved bundling is tractable.

**Documentation architecture**

| Method | What it is / when to use it | Difficulty | Implementation notes and validation target |
|---|---|---|---|
| Diataxis documentation architecture | All docs in four quadrants — tutorials, how-to guides, reference, explanation — plus a fifth pillar: the replication gallery. | Medium | MkDocs-Material or Sphinx+pydata; all tutorial/how-to pages are jupytext-paired notebooks executed in CI on every merge (myst-nb) — failed execution fails the build. Reference auto-generated from numpydoc docstrings, schema-linted in CI. Explanation pages host the theory ("why HAC", "what identification means") at first-year-PhD level. Reference: Procida's Diataxis framework (diataxis.fr). Validate by user-testing: give a grad student a task and watch which quadrant they reach for. |
| Model cards for every estimator | A standardized one-page card per model: assumptions, when to use / when NOT to use, failure modes, required diagnostics, worked example, canonical references, alternatives. Adapted from Mitchell et al. (2019) to econometrics. | Medium | Enforce via schema: structured YAML frontmatter (assumptions[], failure_modes[], see_instead[], references[]) validated in CI so no estimator ships without a complete card; cards render as docs pages and as `model.card()` in the REPL. The "when NOT to use" section is the differentiator (e.g., the ADF card points to Zivot-Andrews / Ng-Perron under breaks and small T). BEAR's technical guide is the rigor benchmark. Every card opens with a one-line load of a canonical bundled dataset. Cheap to mandate early, impossible to retrofit. |
| Executable replication gallery (docs = validation suite) | Notebooks reproducing famous published results, doubling as regression tests: each page asserts key numbers against the paper within tolerance. Marketing, pedagogy, and correctness in one artifact. | High | V1 target list: Ramey-Zubairy (2018 JPE) LP-IV multipliers; Gertler-Karadi (2015) proxy-SVAR; Primiceri (2005) TVP-VAR-SV via the Del Negro-Primiceri (2015) corrected algorithm — document that the correction, not the original ordering bug, is replicated; Kilian (2009) oil SVAR; Giannone-Lenza-Primiceri (2015) BVAR; Uhlig (2005) sign restrictions; Blanchard-Quah (1989); Hamilton (1989) MS-AR; Stock-Watson factor forecasting; Bańbura-Modugno (2014) nowcast news. Traps: archive exact data vintages with hashes and licenses; keep an errata note per page; assert with loose-but-meaningful tolerances (posterior medians within MC error) and fixed seeds; check redistribution rights per dataset. |
| Cookbook / how-to recipes | Short single-task pages ("wild-bootstrap IRF bands", "Driscoll-Kraay SEs for panel LP", "conditional forecast under a fixed policy path", "export a table to LaTeX", "10,000 parallel Monte Carlo replications with a fixed seed"). | Low | The highest-traffic docs in every successful library (pandas cookbook, matplotlib gallery). Rules: ≤30 lines of code, CI-executed, one task per page, cross-linked to model card and explanation page. Seed the first 50 recipes from Stack Overflow's most-asked statsmodels.tsa questions and R-help vars/rugarch threads — literal demand data. |
| Migration guides (from statsmodels, R, Stata/Matlab, pmdarima) | Side-by-side before/after code for the top workflows per origin ecosystem, plus behavior-difference tables (defaults, DoF corrections, missing-data policy). | Medium | The behavior-difference table is the hard, valuable part: enumerate every differing default (statsmodels SARIMAX trend and diffuse-init defaults; vars' df-adjusted sigma; Stata's manual newey lag selection) so migrating users' numbers changing is documented, not alarming. Compatibility shim decision: recommend NO shim (maintenance trap) — exact recipes instead. |
| Error messages and warnings that teach | Every error/warning carries what happened, why it likely happened, what to try, and a short docs URL (e.g., GARCH hitting the alpha+beta=1 bound suggests IGARCH/breaks and concrete next steps). | Medium | Typed hierarchy (NearUnitRootWarning, WeakInstrumentWarning, SingularCovarianceError, NonInvertibleMAWarning) with stable short-links (tslib.dev/e/E042). Rust core returns structured error codes; the Python layer owns the prose. |

**API and UX**

| Method | What it is / when to use it | Difficulty | Implementation notes and validation target |
|---|---|---|---|
| Unified model grammar: Spec → fit()/sample() → Results | One consistent pattern for every estimator: immutable spec object, `.fit()` (frequentist) or `.sample()` (Bayesian) taking data, returning a rich immutable Results object with uniform accessors (`.params`, `.conf_int()`, `.forecast()`, `.irf()`, `.summary()`, `.diagnostics()`). | High | Lock via public RFC before v0.1 (API churn is the documented killer — see sktime): data-in-fit (enables spec reuse across Monte Carlo draws — our core use case); `lags` everywhere, never maxlag/nlags/p mixed; keyword-only after the first argument; a uniform IRF object across VAR/LP/BVAR/state-space so they overlay in one plot (the Stata `irf` insight); Bayesian results expose `.to_arviz()`. Contract-test every estimator against the grammar (sktime's check_estimator idea). Prototype on three deliberately dissimilar families (SARIMA, LP, BVAR) before generalizing — a grammar designed on one family never survives contact with the others. |
| Sensible loud defaults policy | Defaults encode best current practice and say so: LP defaults to lag-augmented inference (Montiel Olea & Plagborg-Møller 2021) with EHW SEs; VAR IRF bands default to bootstrap; GARCH defaults to Bollerslev-Wooldridge robust SEs; HAC bandwidth defaults to a cited automatic rule. | Medium | Policy document enforced in review: every default carries (a) a citation, (b) a docs anchor, (c) a defaults-table entry, (d) a `.summary()` footer note for nonobvious choices ("bands: wild bootstrap, 1000 reps, Gonçalves-Kilian 2004"). Warn once with a reason when a user's choice is detectably dominated (e.g., asymptotic bands with T=80 near a unit root). Never change a default without a deprecation cycle and a changelog showing before/after numbers on a reference dataset. |
| Rich results objects with publication-ready output | Results render as readable `.summary()` text, notebook HTML, booktabs LaTeX (`.to_latex()`), and typed data (`.params` as Series, `.to_arrow()`/`.to_polars()`); estimation metadata (seed, versions, wall time, convergence) rides along. | Medium | Keep the human-readable summary string OUT of the API contract (users will regex it regardless — version the format and say so). LaTeX export needs escaping, a significance-star policy with a no-stars option, and rounding tied to SE magnitude; compile LaTeX output in CI. Support multi-model comparison tables (the modelsummary/stargazer feature with no good Python econometrics peer). |
| Plotting conventions: IRF panels, fan charts, decomposition bars | First-class consistent plots — IRF small-multiple grids with bands, BoE-style fan charts, historical-decomposition stacked bars, FEVD area charts — all returning matplotlib Figure/Axes. | Medium | Per architecture ruling: tidy-data export twins (`.irf_frame()`) plus optional matplotlib rendering; matplotlib is the only hard plotting dependency. Colorblind-safe, grayscale-printable style sheet. IRF conventions to get right: shock-in-columns vs response-in-rows labeling, unit vs one-SD shock stated in the title, band coverage in the legend, zero line always drawn. Fan charts use forecast-distribution percentiles, not ±k·SE, when the distribution exists. Adopt the VAR Toolbox's figure layouts — the community's visual dialect. |
| Diagnostic dashboard | One call (`res.diagnostics()` / `res.plot_diagnostics()`) producing the standard post-estimation battery: residual ACF/PACF, Ljung-Box, QQ/density, CUSUM, ARCH-LM, plus model-specific checks (VAR stability circle, MCMC traces/R-hat). | Medium | Each panel annotated with what "bad" looks like and linked to the symptom page; self-contained HTML report export for sharing with coauthors (mirrors the seasonal package's X-13 inspection GUI). Bayesian diagnostics delegate to ArviZ — rank-normalized split R-hat (Vehtari et al. 2021), not Gelman-Rubin 1992, a common staleness bug elsewhere. |
| Rust core + PyO3 binding architecture (UX aspects) | The architecture choices that determine UX: abi3 wheels on all platforms ("pip install", period), zero-copy NumPy views, GIL released during compute, Rust panics mapped to Python exceptions, never aborts. | High | Rust over Fortran primarily for distribution UX: maturin + abi3 gives painless macOS arm64/Windows wheels (Fortran wheels are a support-ticket generator — see scipy's history). Traps: ndarray layout mismatches (accept C and F order; warn rather than silently copy large arrays); keep ALL prose/formatting/pandas handling out of Rust; extended-precision differences across platforms make bit-identical cross-platform results impossible — promise tolerance-level reproducibility and document it. |

## Implementation warnings

- **Do not build on statsmodels internals.** Wrap-and-extend inherits its speed ceiling and private-API churn. Implement independently and maintain a CI parity suite cross-checking overlapping models to tight tolerances, published as a trust artifact.
- **Every "your numbers differ from Stata/R" report is an adoption crisis.** The usual causes are degrees-of-freedom corrections, HAC bandwidth rules (Newey-West 1994 vs Andrews 1991), Johansen critical-value sources, and GARCH recursion initialization. Enumerate all defaults with citations in a machine-readable table BEFORE users find divergences, and test against the table.
- **Replication targets contain bugs.** Primiceri (2005) requires the Del Negro-Primiceri (2015) corrected ordering; Carriero-Clark-Marcellino (2019) has a 2022 corrigendum. Decide and document per gallery page whether the original numbers or the corrected algorithm is replicated — silently doing either invites accusations of error.
- **Auto-ARIMA parity with R forecast is famously elusive** (pmdarima never achieved it). Match the stepwise search, approximation phase, KPSS defaults, and seasonal heuristics exactly, or explicitly document divergence — "close but different orders" reads as broken to switchers.
- **Teaching warnings must never fire inside hot loops.** Per-iteration warnings in a 10,000-rep bootstrap destroy performance and bury signal — aggregate boundary hits and report once; make every advisory warning class-filterable and globally suppressible.
- **Do not promise bit-identical reproducibility across platforms.** BLAS reduction order, SIMD width, and extended-precision differences make it impossible. Promise and verify tolerance-level reproducibility, stamp the tolerance into results, and use fixed-order reductions plus counter-based RNG so results are identical across thread counts on one platform.
- **Seed parallel workers per-task, not per-thread.** Per-thread seeding makes bootstrap results depend on scheduler timing — key RNG substreams by replication index (Philox/SeedSequence) and add a CI test asserting identical CIs at 1, 4, and 16 threads.
- **The `.summary()` string will be regex-parsed no matter what you say.** Keep machine-readable accessors complete so users do not need to, version the text format, and never let downstream numbers exist only in formatted output.
- **LaTeX export is a bug farm.** Escape user variable names, tie displayed precision to standard-error magnitude, make significance stars optional (journal policies diverge), and compile the output in CI — a table that breaks pdflatex at a submission deadline is a user lost forever.
- **API churn is the documented killer in this space** (sktime's reworks; rugarch's author rewriting his own API). Lock the estimator grammar via a public RFC prototyped on three dissimilar families (SARIMA, LP, BVAR) before v0.1, then enforce two-release deprecation cycles with CI checks on public symbols.
- **Benchmark marketing backfires if configurations are unequal** (the Nixtla/pmdarima dispute). Verify identical logliks/estimates before timing, publish runner scripts and hardware, and include losing benchmarks — credibility is the product.
- **CI-executed docs with MCMC content flake.** Fix seeds, use small advertised-as-small draw counts with a "scale up in practice" banner, cache expensive artifacts keyed on API version, and assert paper-replication numbers with tolerances calibrated to Monte Carlo error, not machine epsilon.
- **Dual pandas/polars support founders on time semantics, not data conversion.** Polars has no frequency-aware index — define an internal time-index contract (explicit frequency, validated on ingest, loud errors on irregular spacing) rather than inferring silently from either ecosystem.
- **Mixed-frequency alignment is the classic silent MIDAS bug** (an off-by-one in high-to-low frequency mapping shifts results materially). Make alignment explicit, printable, and unit-tested against midasr vignette outputs.
- **Fortran-based wheels are a distribution tax** (gfortran runtime shipping, macOS arm64 pain — see scipy's history). Choosing Rust instead, budget for hand-maintained .pyi stubs and safe GIL/callback patterns — progress callbacks from released-GIL threads deadlock if done naively.
- **Dataset redistribution rights vary.** FRED terms permit redistribution with attribution, but some journal replication files and commercial series do not — audit licenses per gallery dataset and ship download-on-first-use fetchers with hashes where redistribution is barred.
- **Beware stale statistics in incumbent code you validate against.** Use rank-normalized split R-hat (Vehtari et al. 2021), not Gelman-Rubin 1992; use modern response-surface bounds-test p-values (Kripfganz-Schneider 2020), not the original PSS tables. Matching an incumbent can mean matching its obsolescence — validate against the literature first, packages second.
